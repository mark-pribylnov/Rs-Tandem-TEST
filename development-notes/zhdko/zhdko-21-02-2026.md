# Дата: 21.02.2026

**Описание**  
Писал компонент тостера. Просмотрел несколько вариантов реализации: везде используется @Injectable. В тех компонентах, которые писал ранее, такое еще не применял, поэтому пошел читать документацию. Концепт оказался достаточно простым: он позволяет создать синглтон, который можно использовать в любом месте с помощью inject.
Появился вопрос: как правильно группировать утилиты для приложения? Создавать отдельный сервис для каждой мелочи кажется избыточным. Прочитал гайд от Angular «Defining dependency providers». Тема сложная для понимания с первого раза, но важная, так как затрагивает архитектуру API. На данном этапе решил не закапываться в провайдеры и сосредоточиться на логике тостера.

**Проблемы**

1. Организация связи между сервисом (логикой) и компонентом тостера (визуалом).
2. Реализация вывода сразу нескольких уведомлений в одном стеке.
3. Упорядочивание уведомлений: новые должны быть внизу, а старые уходить вверх.

**Решение**

1. Изучая примеры, сначала попробовал написать логику на BehaviorSubject, но в процессе понял, что на сигналах это выглядит чище, и переписал все на signal.
2. Для отображения очереди использовал массив объектов в сигнале. Чтобы Angular правильно обрабатывал анимации и не путал сообщения, реализовал уникальные ID для каждого тоста.
3. Для автоматического скролла к новым сообщениям использовал новый viewChild и эффект (effect), который следит за изменениями в массиве.

**Планы**

- Настроить более плавную анимацию появления и удаления.
